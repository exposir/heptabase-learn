## 🎯 项目定位

这不是一个普通的技术笔记仓库，而是一个**知识的沉淀场域**，一个**思想的实验室**。

**核心理念**：
- 📚 **深度优于广度**：宁可深挖一口井，不愿浅尝百家泉
- 🔗 **关联优于孤立**：知识的力量在于网络，而非碎片
- 🧠 **哲学优于技术**：理解"为什么"比记住"怎么做"更重要
- 🌱 **持续优于完美**：知识如花园，需要持续耕耘而非一次建成

**知识管理工具**：
- **Heptabase**：可视化知识图谱，白板式思维整理
- **Markdown**：纯文本记录，版本可控，永久可读
- **Git**：知识的版本管理，思想的演进轨迹

## 🧩 GEB 分形文档系统

本项目采用 **GEB（Gödel, Escher, Bach）分形文档系统**，实现代码与文档的同构。

### 三层结构

```
┌─────────────────────────────────────┐
│ L1: /CLAUDE.md                      │  ← 项目宪法·全局地图
│ "Heptabase Learn知识库"            │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ L2: 模块/CLAUDE.md                  │  ← 模块地图·成员清单
│ "前端开发的历史与哲学"              │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ L3: 文件头部注释                    │  ← INPUT/OUTPUT/POS契约
│ <!--                                │
│ - [INPUT]: 依赖什么                 │
│ - [OUTPUT]: 提供什么                │
│ - [POS]: 在哪个位置                 │
│ -->                                 │
└─────────────────────────────────────┘
```

### 核心原则

**代码是机器相，文档是语义相，两相必须同构**

- 任何架构变更，必须同步更新文档
- 任何文档变更，必须反映实际结构
- L1/L2/L3 三层文档，分形自相似

详见：[CLAUDE.md](./CLAUDE.md)

**核心协作哲学**：
- **三层认知**：现象层（症状）→ 本质层（根因）→ 哲学层（真理）
- **Good Taste**：优先消除特殊情况，而非增加if/else
- **实用主义**：先写最简单能运行的，再考虑扩展
- **简单性**：函数短小只做一件事，复杂性是最大的敌人

## 📝 写作规范

### 文档标准

**必需元素**：
1. ✅ L3头部契约（INPUT/OUTPUT/POS）
2. ✅ 清晰的章节结构
3. ✅ 代码示例（必须可运行）
4. ✅ 哲学概念锚定
5. ✅ 参考文献

**质量要求**：
- **深度**：专题文章至少8000字
- **哲学密度**：至少30%内容涉及哲学思考
- **实用性**：理论必须配合实践案例
- **可读性**：三层结构（现象→本质→哲学）清晰

### 命名规范

**主文档**：`README.md`
**专题文章**：`[主题]的哲学解构：[副标题].md`
**示例**：
- ✅ `MVC模式的哲学解构：架构思想的深度剖析.md`
- ✅ `Redux的哲学演进：从Flux到时间旅行调试.md`
- ❌ `mvc.md`（太简略）
- ❌ `mvc详解.md`（缺乏哲学视角）
